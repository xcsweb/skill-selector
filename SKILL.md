# Skill 选择器元技能 (skill-selector)

## 核心设计原则（三条铁律）

### 原则一：项目优先（Project-First）

所有 Skill 操作必须基于当前项目上下文进行决策。

| 操作场景 | 项目上下文要求 | 决策依据 |
|---------|--------------|---------|
| Skill 调用 | 必须识别项目类型、技术栈 | 项目配置文件（package.json, tsconfig.json 等） |
| Skill 创建 | 必须符合项目架构规范 | 项目目录结构和代码风格 |
| Skill 更新 | 必须保持与项目一致性 | 项目依赖版本和 API 兼容性 |
| Skill 删除 | 必须评估项目影响范围 | 项目引用关系和依赖图 |

### 原则二：效率优先（Efficiency-First）

**5 级优先链（从高到低）：**

1. **P0 - 缓存命中**：直接读取项目缓存，零 token 消耗
2. **P1 - 规则匹配**：基于预定义规则快速匹配，低 token 消耗
3. **P2 - 轻量分析**：读取关键文件进行简单分析，中等 token 消耗
4. **P3 - 深度分析**：全量扫描项目代码库，高 token 消耗
5. **P4 - 交互确认**：需要用户参与决策，最高 token 消耗

### 原则三：全局保护（Global-Protection）

**3 条安全边界：**

1. **禁止修改系统 Skill**：不得修改 `.trae/skills/system/` 目录下的任何 Skill
2. **禁止破坏性操作**：删除/更新操作前必须备份，并提供回滚方案
3. **禁止循环依赖**：Skill 之间不允许形成依赖环，检测到立即中断

### 缓存产物选型对比

| 方案 | 格式 | 优点 | 缺点 | 推荐度 |
|-----|------|------|------|-------|
| **A** | Markdown | 可读性好、易维护、支持注释 | 解析需额外处理 | ⭐⭐⭐⭐⭐ |
| B | JSON | 结构化强、程序友好 | 可读性差、不支持注释 | ⭐⭐⭐ |

### Scenario 说明

#### Scenario 1：首次使用
- **触发条件**：`profiles/{project-name}.md` 不存在
- **执行流程**：Bootstrap 全量分析 → 生成缓存 → 返回结果
- **Token 消耗**：高（P3 级别）
- **耗时**：{default: 10-30秒}

#### Scenario 2：后续使用
- **触发条件**：缓存文件存在且未过期
- **执行流程**：读取缓存 → 快速匹配 → 返回结果
- **Token 消耗**：极低（P0 级别）
- **耗时**：{default: <1秒}

#### Scenario 3：缓存回退
- **触发条件**：缓存文件损坏或解析失败
- **执行流程**：删除损坏缓存 → Bootstrap 重建 → 返回结果
- **Token 消耗**：中（P2 级别）
- **日志级别**：WARNING

#### Scenario 4：缓存刷新
- **触发条件**：用户显式请求或项目重大变更
- **执行流程**：删除旧缓存 → Bootstrap 全量分析 → 生成新缓存
- **Token 消耗**：高（P3 级别）
- **建议频率**：{default: 每周一次或项目大版本更新后}

## 触发决策框架（双模式）

### 触发模式总览表

| 模式 | 触发方式 | 适用场景 | Token 成本 | 准确度 |
|-----|---------|---------|-----------|--------|
| 手动触发 | 用户明确指令 | 特定需求、调试 | 高 | 100% |
| 自动触发 | 语义识别 | 常规开发流程 | 低 | 95%+ |

### 手动触发规则

**关键词匹配（不区分大小写）：**
- `use skill` / `调用 skill` / `使用 skill` / `选 skill` / `选择 skill`
- `create skill` / `创建 skill` / `新建 skill`
- `update skill` / `更新 skill` / `修改 skill`
- `delete skill` / `删除 skill` / `移除 skill`
- `list skills` / `列出 skills` / `查看 skills`
- `check skill` / `检查 skill` / `skill 健康`
- `refresh` / `刷新` / `rebuild` （缓存操作）
- `dedup` / `去重` / `重复` （精简分析）

**[Bug #1 兼容性提示机制]**
当检测到手动触发时，如果存在多个匹配的 Skill：
1. 输出兼容性提示："检测到多个可能匹配的 Skill，请确认具体需求"
2. 列出候选 Skill 及其置信度
3. 等待用户明确选择后再执行

### 自动触发规则

**优先检查缓存：**

```python
def should_trigger(user_input: str, cache_exists: bool) -> bool:
    if not cache_exists:
        return True  # 无缓存，需 Bootstrap
    
    # 缓存存在，快速匹配
    keywords = extract_keywords(user_input)
    cached_skills = load_cache()
    
    for skill in cached_skills:
        if match_keywords(keywords, skill['keywords']):
            return True
    
    return False
```

## 匹配优先级算法

### P0-P4 五级优先级表

| 优先级 | 名称 | 触发条件 | Token 成本 | 响应时间 |
|-------|------|---------|-----------|---------|
| **P0** | 手动指定 | 用户明确指定 Skill 名称 | 极低 | 即时 |
| **P1** | 精确匹配 | 关键词完全匹配 Skill 描述 | 低 | <1s |
| **P2** | 语义相似 | 语义向量相似度 > {default: 0.85} | 中 | 1-3s |
| **P3** | 上下文推断 | 基于项目上下文和历史行为 | 中高 | 3-5s |
| **P4** | 模糊猜测 | 无明确匹配时的最佳猜测 | 高 | >5s |

### 同类型多 Skill 决策示例

**场景**：用户请求"处理日期格式化"

| 候选 Skill | 匹配度 | 推荐理由 | 最终选择 |
|-----------|--------|---------|---------|
| date-formatter | 0.95 | 专门处理日期格式化 | ✅ 首选 |
| utils-common | 0.72 | 通用工具集，含日期功能 | ❌ 过于宽泛 |
| i18n-handler | 0.65 | 国际化相关，非核心功能 | ❌ 偏离主题 |

### [Bug #1 已修复] 冲突解决规则

**6 条冲突解决规则（按优先级排序）：**

1. **P0 手动 > 一切**：用户显式指定的 Skill 优先级最高
   - **修复**：但输出兼容性提示，提醒潜在冲突
   - 示例："您选择了 X Skill，但 Y Skill 也可能适用，是否需要对比？"

2. **精确匹配 > 语义匹配**：关键词完全匹配优于相似度计算

3. **专用 > 通用**：专门用途的 Skill 优于通用 Skill

4. **高频使用 > 低频使用**：历史使用频率高的优先

5. **近期更新 > 陈旧版本**：最近更新的 Skill 优先

6. **项目定制 > 默认提供**：项目专属 Skill 优于全局 Skill

## 操作优先级统一管理体系

### 操作类型与优先级矩阵

| 操作类型 | 优先级 | 前置条件 | 回滚能力 | 影响范围 |
|---------|-------|---------|---------|---------|
| **查询** | P0 | 无 | N/A | 只读 |
| **创建** | P1 | 命名唯一性检查 | 支持（删除） | 新增资源 |
| **更新** | P2 | 版本锁定 + 备份 | 支持（恢复备份） | 修改现有资源 |
| **删除** | P3 | 引用检查 + 确认 | 支持（从回收站恢复） | 移除资源 |
| **批量操作** | P4 | 单项操作的并集 | 部分支持 | 多个资源 |

### 多操作并发规则

1. **同资源互斥**：同一 Skill 不能同时被多个操作锁定
2. **读写分离**：查询可与写操作并发，写操作之间串行
3. **原子性保证**：批量操作要么全部成功，要么全部回滚
4. **超时机制**：单个操作超时时间 {default: 30秒}，批量操作 {default: 120秒}

### Scenario 示例

#### Scenario 1：同时需要调用和创建
**场景**：用户请求"调用 auth skill，如果没有就创建一个"
**处理流程**：
1. 先执行查询（P0）→ 检查是否存在
2. 若存在 → 直接调用
3. 若不存在 → 创建（P1）→ 调用
4. 保证原子性：两步操作在同一事务中完成

#### Scenario 2：更新 vs 删除冲突
**场景**：用户 A 正在更新 skill-v1，用户 B 请求删除 skill-v1
**处理流程**：
1. 检测到锁冲突
2. 返回错误："Skill 正在被其他操作占用，请稍后重试"
3. 建议：等待 {default: 5秒} 后重试，或联系操作者协调

## Token 优化 — 脚本化轻量机制

### 问题分析

传统 Skill 调用的 Token 消耗：

| 阶段 | Token 消耗 | 占比 | 优化空间 |
|-----|----------|------|---------|
| Skill 加载 | 500-2000 | 30% | ⭐⭐⭐⭐⭐ |
| 上下文理解 | 1000-3000 | 40% | ⭐⭐⭐ |
| 参数解析 | 500-1500 | 20% | ⭐⭐⭐⭐ |
| 结果返回 | 200-800 | 10% | ⭐⭐ |

### 脚本化分层结构图

```
┌─────────────────────────────────────────────┐
│              传统模式（重）                   │
│  完整 SKILL.md (2000-5000 tokens)           │
│  + 动态上下文 (1000-3000 tokens)            │
│  = 总计 3000-8000 tokens/次                 │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│              脚本化模式（轻）                 │
│  轻量索引 (200-500 tokens)                  │
│  + 按需加载模块 (300-1000 tokens)           │
│  = 总计 500-1500 tokens/次                  │
│  📉 节省 60-80% tokens                      │
└─────────────────────────────────────────────┘
```

### 改造标准表

| 改造项 | 标准 | 阈值（default） | 优化效果 |
|-------|------|---------------|---------|
| 索引文件大小 | ≤ 500 行 | {default: 400行} | 减少 70% 加载 token |
| 模块粒度 | 单一职责 | 每模块 ≤ 200 行 | 按需加载减少 50% |
| 动态导入 | 延迟加载 | 首次访问时加载 | 启动速度提升 3x |
| 缓存策略 | LRU 缓存 | 缓存 {default: 10个} 模块 | 重复调用零消耗 |

## 精简去重

### 删除安全准则（4 步流程）


### 功能覆盖检测算法

```python
def check_coverage(skill_a: dict, skill_b: dict) -> float:
    """
    计算 Skill B 对 Skill A 的功能覆盖率
    返回值：0.0 - 1.0（0% - 100%）
    """
    features_a = set(extract_features(skill_a))
    features_b = set(extract_features(skill_b))
    
    if not features_a:
        return 0.0
    
    coverage = len(features_a & features_b) / len(features_a)
    return coverage
```

### [Bug #2] 去重决策矩阵

| 情况 | 覆盖度 | 决策建议 | 操作 |
|-----|-------|---------|------|
| A ⊇ B 且 A 更优 | ≥95% | 删除 B，保留 A | 安全删除 |
| A ≈ B（高度重叠） | 90-95% | 合并为一个 Skill | 合并重构 |
| A ∩ B 部分重叠 | <90% | 两者都保留 | 保留双 Skill |
| **B 与项目相关** | **任意** | **保留 B** | **不删除** |
| A 新版，B 旧版 | ≥95% | 删除旧版 B | 版本升级 |
| A 和 B 都废弃 | <50% | 双双删除 | 批量清理 |

**覆盖度阈值三级制（已修复 Bug #2）：**

| 覆盖度范围 | 含义 | 处理策略 |
|-----------|------|---------|
| **≥95%** | 完全覆盖 | 可安全删除冗余 Skill |
| **90%-95%** | 高度重叠 | 建议合并或重构 |
| **<90%** | 部分覆盖 | 保留各自特色功能 |

```python
    """
    检测两个 Skill 之间的互斥关系

    """
    results = {}

    if func_similarity >= 0.95:  # [Bug #2 修复] 阈值 ≥95%

            'confidence': func_similarity,

        }

    if shared_resources and has_write_conflict(shared_resources):

            'type': '资源竞争',

        }

    if has_circular_dependency(skill_a, skill_b):
        results['EX-03'] = {
            'type': '循环依赖',
            'confidence': 1.0,
            'details': '检测到依赖环路'
        }

    # EX-04: 配置冲突检测
    config_conflicts = check_config_conflicts(skill_a, skill_b)
    if config_conflicts:
        results['EX-04'] = {
            'type': '配置冲突',
```

### Scenario 示例

#### Scenario 1：应自主创建
**对话片段**：
```
用户：帮我写个用户验证函数
AI：好的，这是验证函数代码...
用户：再来一个类似的，但是针对管理员
AI：好的，这是管理员验证函数...
用户：又来一个，编辑者的验证
用户：（第5次类似请求）每次都写这些验证好麻烦啊
```
**检测分析**：
- ✅ 条件1：重复请求 5 次（>5次/10轮）
- ✅ 条件2：明确表达痛点（"好麻烦"）
- ✅ 条件3：可抽象为通用模式（角色权限验证）
- ✅ 条件4：项目中已有 auth 相关 Skill 可参考
- ✅ 条件5：创建成本低（约 300 tokens）
- **触发计数**：5/5 ≥ 3
- **结论**：🟢 触发自主创建，提议创建 `auth-role-validator`

#### Scenario 2：不应自主创建
**对话片段**：
```
用户：帮我写个快速排序算法
AI：好的，这是快速排序实现...
用户：再写个归并排序
AI：好的...
用户：这两个算法哪个更快？
```
**检测分析**：
- ❌ 条件1：仅 2 次请求（<5次）
- ❌ 条件2：无明确痛点表达
- ✅ 条件3：可抽象为排序算法集合
- ❌ 条件4：项目是业务系统，非算法库
- ❌ 条件5：创建成本中等（约 800 tokens）
- **触发计数**：2/5 < 3
- **结论**：🔴 不触发，继续正常对话

### 生命周期关键节点说明

1. **创建阶段**：
   - 必须通过三层评估模型
   - 生成完整的 SKILL.md 文件
   - **立即同步缓存**，确保后续调用可命中

2. **活跃使用阶段**：
   - 正常调用无需缓存操作
   - 仅在性能异常或错误时触发缓存更新

3. **健康检查阶段**：
   - 定期执行（建议频率：{default: 每周一次}）
   - 检测依赖变化和版本过时
   - 发现问题时及时同步缓存状态

4. **废弃处理阶段**：
   - 严格遵循废弃判定标准
   - 归档或删除后必须同步缓存
   - 避免已废弃 Skill 被继续调用

5. **缓存同步节点**：
   - 创建后：新增节点到缓存
   - 更新后：修改缓存中的 Skill 信息
   - 废弃后：标记或移除缓存节点
   - 刷新时：全量重建缓存文件
