# skill-selector

**Skill 选择与管理元技能（Meta-Skill）**——让 AI 自己管理自己的 Skill。

当一个项目积累了大量 Skill 后，AI 该怎么选？该不该新建？旧的要不要删？skill-selector 给了 AI 一套完整的决策框架。

## 核心能力

- **智能匹配**：P0-P4 五级优先级算法，不再盲目试错
- **精简去重**：三级覆盖度阈值（≥95% 实质等同 / 90-95% 灰色地带 / <90% 各有特色）
- **废弃检测**：8 种触发条件，含大模型过时淘汰机制
- **互斥校验**：5 种互斥类型自动检测
- **自主创建**：对话中检测重复模式，主动建议封装新 Skill
- **Token 优化**：Bootstrap → Cache 两层架构，后续调用节省约 95% token

## 三条设计铁律

1. **项目优先** — 跟当前项目有关的优先
2. **效率优先** — Token 最省 > 执行最快 > 效率最高 > 功能完整 > 通用性强
3. **全局保护** — 全局 Skill 可能被其他项目依赖，不误删

## 快速开始

### 安装（30 秒）

将 `skill-selector` 目录复制到全局 Skills 目录：

```
~/.agents/skills/skill-selector/
├── SKILL.md              # 核心元技能
└── profiles/
    └── _template.md      # 项目缓存模板
```

复制即用，无需配置环境变量或安装依赖。

## 使用指南

### 一、首次使用 —— Bootstrap 阶段

在新项目中第一次触发 skill-selector 时（例如对 AI 说"帮我选一个合适的 Skill"），它会自动完成以下操作：

1. **扫描全部 Skill**：读取全局和项目级别的所有 Skill 描述信息
2. **去重分析**：计算功能覆盖度，识别冗余 Skill
3. **互斥检测**：检查 5 类冲突关系
4. **Token 评估**：统计每个 Skill 的 token 消耗权重
5. **生成缓存文件**：在项目的 `.trae/` 目录下生成 `skill-selector-cache.md`

> Bootstrap 只需执行一次，耗时约 10-30 秒（取决于 Skill 数量）。完成后会提示："已为您生成项目缓存，后续将优先使用缓存（节省约 95% token）"。

### 二、日常使用 —— Cache 阶段

Bootstrap 完成后，每次涉及 Skill 选择时直接读缓存：

| 操作方式 | 说明 | Token 消耗 |
|---------|------|-----------|
| 自动匹配 | AI 根据任务自动从缓存选择最佳 Skill | ~200 tokens |
| 手动查询 | 对 AI 说"列出当前项目可用的 Skill" | ~200 tokens |
| 刷新缓存 | 对 AI 说"刷新 skill 缓存" | ~5000 tokens（全量重建） |

### 三、手动触发指令速查

| 你想做什么 | 对 AI 说 | 触发模式 | 命中关键词 |
|-----------|--------|---------|-----------|
| 选一个 Skill | "帮我选一个合适的 Skill 处理这个任务" | 手动 | `选` + `Skill` |
| 查看所有 Skill | "列出当前项目可用的 Skill" | 手动 | `列出` + `Skill` |
| 创建新 Skill | "我想创建一个 xxx 的 Skill" | 手动 | `创建` + `Skill` |
| 删除 Skill | "删除 xxx 这个 Skill" | 手动（需确认） | `删除` + `Skill` |
| 检查 Skill 健康 | "检查一下我的 Skill 有没有问题" | 手动 | `检查` + `Skill` |
| 刷新缓存 | "刷新 skill 缓存" | 手动 | `刷新` |
| 去重分析 | "看看有没有重复的 Skill" | 手动 | `重复` + `Skill` |

> 以上 7 条指令均命中 SKILL.md 定义的手动触发关键词，可直接触发 skill-selector。
> 此外，当用户输入未包含关键词但语义上涉及 Skill 选择时，AI 也可通过自动触发规则（缓存关键词语义匹配）自行判断是否调用。

### 四、理解匹配优先级（P0-P4）

| 优先级 | 名称 | 含义 | 示例 |
|-------|------|------|------|
| **P0** | 手动指定 | 用户明确说了用哪个 | "用 lark-doc 处理这个文档" |
| **P1** | 精确匹配 | 关键词完全命中 | 任务含 "飞书文档" → lark-doc |
| **P2** | 语义相似 | 语义向量相似度高 | "处理在线协作文档" → lark-doc |
| **P3** | 上下文推断 | 基于项目上下文推测 | 在 Vue 项目中 → web-dev |
| **P4** | 模糊猜测 | 无明确匹配时的兜底 | 最后的备选 |

### 五、理解创建评估（三层模型）

当 skill-selector 建议创建新 Skill 时，它经过了三层评估：

```
第一层：项目价值（否决项）
  ├── 是否解决重复性问题？（出现 ≥3 次）
  ├── 是否提升团队效率？（节省 ≥30% 时间）
  └── 是否降低出错率？（减少 ≥50% 错误）
      ↓ 全部通过
第二层：效率指标
  ├── Token 消耗节省率？（≥40%）
  ├── 执行时间缩短率？（≥25%）
  └── 维护成本降低率？（≥20%）
      ↓ 全部通过
第三层：可行性
  ├── 技术实现复杂度？（≤中等）
  ├── 与现有架构兼容？（100% 兼容）
  └── 团队接受度预估？（≥80%）
```

只有三层全部通过才会建议创建。任何一层不通过都会给出具体原因。

### 六、理解废弃判定（6 维度）

| 维度 | 权重 | 废弃阈值 | 说明 |
|-----|------|---------|------|
| 使用频率 | 30% | < 1次/月 | 最近 90 天调用次数 |
| 替代性 | 25% | ≥85% 被覆盖 | 功能被其他 Skill 取代 |
| 维护状态 | 20% | 停止维护 > 180天 | 最后更新时间 |
| 文档完整性 | 10% | < 60% 字段填写 | SKILL.md 完整度 |
| 测试健康度 | 10% | < 70% 通过率 | 测试通过情况 |
| 大模型过时 | 5% | 落后 ≥2 个主版本 | LLM 已原生支持该能力 |

满足足够多维度阈值后标记为废弃，不会立即删除，先进入观察期。

### 七、常见场景示例

**场景 1：处理飞书文档**

> 你："帮我处理一下这个飞书文档"
>
> AI（经过 skill-selector）：读取缓存 → 匹配规则"飞书 Docx token → lark-doc" → 直接调用 lark-doc ✅ （1 步到位）

**场景 2：发现冗余 Skill**

> 你："检查一下我的 Skill 有没有问题"
>
> AI（经过 skill-selector）：扫描发现 `lark-doc-legacy` 与 `lark-doc` 重叠度 95%，legacy 与当前项目无关 → 推荐安全删除 legacy

**场景 3：大模型过时淘汰**

> 你："检查 Skill 健康状态"
>
> AI（经过 skill-selector）：检测到 `json-formatter` 功能已被 LLM v4.0+ 原生支持 → 标记为 superseded-by-llm-v4 → 建议确认无依赖后删除，每次省 ~300 tokens

**场景 4：自主创建建议**

> 你：（第 5 次让 AI 写类似的验证函数）"每次都写这些验证好麻烦啊"
>
> AI（经过 skill-selector）：检测到重复模式 ≥3 次 + 明确痛点表达 → 建议创建 `request-validator` Skill → 征求你的确认
